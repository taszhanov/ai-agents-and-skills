# Data Dictionary Writer

You are creating a data dictionary for a business analyst. A data dictionary is a reference document that explains what data exists in a system — what each table and field is for, what values are allowed, and how fields relate to each other.

Think of it as a glossary for your database.

## What you need before starting

Ask for (if not provided):
1. **Tables or entities to document** — which tables or collections?
2. **Schema or sample data** — paste a table definition, a sample row, or a screenshot of the structure
3. **Business context** — what system is this from? What process does it support?
4. **Audience** — is this for developers, analysts, or business stakeholders?

---

## Output structure

### Overview

| Field | Value |
|-------|-------|
| **System / Database** | [e.g. CRM, Order Management, HR System] |
| **Documented by** | [Name / Role] |
| **Date** | [Today's date] |
| **Version** | [e.g. 1.0] |

**Purpose:** 2–3 sentences explaining what this database/system is for and what business process it supports.

---

### Table: [table_name]

**Business description:** What does this table represent in plain English? (e.g. "Stores one record per customer order. Each row is a single transaction.")

**Used by:** Which systems, reports, or processes read from or write to this table?

| Column Name | Data Type | Required? | Description | Example Value | Allowed Values / Rules |
|-------------|-----------|-----------|-------------|---------------|------------------------|
| `id` | INTEGER | Yes | Unique identifier for each record. Auto-generated by the database. | 10432 | Any positive integer |
| `status` | VARCHAR | Yes | Current state of the record in the workflow. | "active" | "active", "inactive", "pending", "archived" |
| `created_at` | TIMESTAMP | Yes | Date and time the record was created. Stored in UTC. | 2024-03-15 09:42:00 | Any valid datetime |
| `customer_id` | INTEGER | Yes | Links to the `customers` table — identifies which customer this belongs to. | 8821 | Must exist in `customers.id` |

**Notes:** Any additional rules, exceptions, or known issues with this table's data quality.

---

### Relationships

If multiple tables are documented, add a relationships section:

| Table | Relates to | How | Description |
|-------|------------|-----|-------------|
| `orders` | `customers` | `orders.customer_id = customers.id` | Each order belongs to one customer. One customer can have many orders. |
| `orders` | `order_items` | `order_items.order_id = orders.id` | Each order has one or more line items. |

---

### Lookup / Reference Values

For any field that uses a fixed set of codes or statuses, document what each value means:

**`orders.status`**

| Value | Plain-English Meaning | When is this set? |
|-------|-----------------------|-------------------|
| `pending` | Order has been placed but payment not yet confirmed | Immediately on order creation |
| `confirmed` | Payment received, order is being processed | When payment gateway confirms |
| `shipped` | Order has left the warehouse | When a tracking number is assigned |
| `cancelled` | Order was cancelled before shipping | By customer or support agent |

## After writing

Tell the user:
- How many tables / fields were documented
- Any fields where the meaning was unclear and requires confirmation from a developer or system owner
- Recommended next step (e.g. "Share with the development team for review before finalising")
